# v1.2 Runbook：训练、评估与证据包

保存日期：2026-06-22

本文是 `docs/v1.2-implementation-plan.md` 的执行手册，目标是让 v1.2-a 训练和后续矩阵评估能按同一套证据链复盘。

## 1. 训练前检查

v1.2-a 默认先打 `common_ai`，用于隔离验证终局奖励、推塔窗口特征和学习率下调。

确认关键配置：

```bash
python3 utils/v1_2_preflight.py --md logs/v1.2/preflight.md --csv logs/v1.2/preflight.csv
python3 -m unittest tests/test_train_env_conf.py tests/test_ppo_optimization.py
python3 utils/offline_sync.py check --preset v1.2
```

应满足：

- `agent_ppo/conf/train_env_conf.toml` 中 `opponent_agent = "common_ai"`。
- `eval_opponent_type = "common_ai"`。
- `Config.INIT_LEARNING_RATE_START = 5e-4`。
- `GAMMA/BETA/clip/grad_clip` 保持 v1.1 后的保守设置。
- v1.2 同步包 preset 包含训练、评估、证据工具、测试和 runbook 必备文件。
- `preflight.md/csv` 没有 `FAIL` 项。
- `launch_manifest.md/json` 记录本次训练的 commit、同步包 hash、环境变量和报告命令。

如需生成离线同步包：

```bash
python3 utils/offline_sync.py pack --preset v1.2 --note v1.2-a-ready -o sync_package.txt
python3 utils/v1_2_launch_manifest.py --sync-package sync_package.txt --run-id v1.2-a-001 --md logs/v1.2/launch_manifest.md --json logs/v1.2/launch_manifest.json
python3 utils/v1_2_experiment_plan.py --stage v1.2-a --json logs/v1.2/experiment_plan.json --md logs/v1.2/experiment_plan.md
```

`experiment_plan.md/json` 固定记录本轮科研主线、消融组、固定评估矩阵、成功指标和报告比较命令；后续 `v1.2`、`no_window_reward`、`no_terminal_reward`、`death_only_risk` 等训练都应对齐这份方案，避免不同实验使用不同评估口径。
如果本轮需要覆盖 reward 权重或课程比例，应优先通过 `v1_2_launch_manifest.py --reward-weight-overrides ... --opponent-schedule ...` 写入 launch manifest，再按 manifest 中的环境变量启动训练。

训练时保留默认训练账本：

```bash
export HOK_TRAINING_RECORDER=1
export HOK_TRAINING_RECORD_DIR=logs/run_records/v1.2-a
export HOK_TRAINING_RUN_ID=v1.2-a-001
export HOK_REWARD_PROFILE=v1.2
```

`HOK_REWARD_PROFILE` 会写入训练账本。默认 `v1.2` 不改变当前权重；消融实验可用 `no_window_reward`、`no_terminal_reward`、`death_only_risk`。需要精确覆盖单项权重时，可额外设置：

```bash
export HOK_REWARD_WEIGHT_OVERRIDES=death:5,push_window_tower_damage:0
```

## 2. v1.2-a 训练

平台训练使用当前 `agent_ppo/conf/train_env_conf.toml`。这一轮不要先切 `selfplay`，否则很难判断 reward/feature 改动本身是否有效。

训练观察重点：

- common_ai 胜率是否超过 v1.1 `step-15000` 的 0.84，或胜率接近但死亡显著下降。
- 敌塔血是否低于 v1.1 `step-15000` 的 1401。
- 死亡是否低于 v1.1 后段 `step-17057` 的 3.09。
- `push_window_tower_damage` 是否上升，`unsafe_dive` 是否下降。
- `push_window_active_frames` 是否足够多，`unsafe_dive_active_frames` 是否随训练下降；这两个字段是 0 权重诊断量，只用于解释推塔窗口和危险越塔频次，不改变训练目标。

建议保留至少三组短训对照：

- `HOK_REWARD_PROFILE=v1.2`：完整 v1.2 reward。
- `HOK_REWARD_PROFILE=no_window_reward`：保留终局和保塔，只去掉窗口 reward。
- `HOK_REWARD_PROFILE=no_terminal_reward`：保留 dense reward，只去掉终局胜负/超时 reward。

多组实验都生成证据包后，可横向比较：

```bash
python3 utils/compare_experiment_reports.py \
  logs/v1.2/report-v1.2 \
  logs/v1.2/report-no-window \
  logs/v1.2/report-no-terminal \
  logs/v1.2/report-death-risk \
  --csv logs/v1.2/report_comparison.csv \
  --md logs/v1.2/report_comparison.md
```

`report_comparison.md/csv` 会输出 `ablation_interpretation` 和 `research_story_verdict`。前者判断消融是否整体支持完整 v1.2 recipe，后者把指标变化翻译为科研叙事结论，例如 `supports_push_window_modeling`、`supports_terminal_alignment` 或 `death_risk_reduces_deaths_but_hurts_objective`。比较多组实验前先确认 `matchup_filter_eval_only`、`matchup_filter_opponent_agent` 和 `candidate_gate_matchup_filter` 一致。

## 3. 训练后生成证据包

训练日志落到 `logs/v1.2` 后，生成本地证据包：

```bash
python3 utils/build_experiment_report.py \
  --log-dir logs/v1.2 \
  --record-dir logs/run_records/v1.2-a \
  --launch-manifest logs/v1.2/launch_manifest.json \
  --experiment-plan logs/v1.2/experiment_plan.json \
  --experiment-name v1.2 \
  --output-dir logs/v1.2/report-v1.2 \
  --checkpoints 15000,17057 \
  --heroes 112,133,199 \
  --repeats 20
```

若要把固定评估矩阵同时展开为召唤师技能网格，在同一命令后追加：

```bash
  --skill-grid \
  --skills 80107,80110,80115
```

核心产物：

- `training_summary.md/csv`：训练 step 级曲线摘要。
- `run_metadata_summary.md/csv`：训练启动配置快照、reward profile、opponent schedule 和关键配置 hash。
- `opponent_curriculum_summary.md/csv`：若有 episode 账本，按实际 `opponent_source` / `opponent_agent` 统计课程采样局数、胜率、死亡和塔血。
- `manifest.md`：绑定 launch manifest 与 experiment plan，记录本报告所属消融组、科研假设和成功指标上下文。
- `matchup_summary.md/csv`：按 checkpoint、matchup、对手聚合。
- `checkpoint_ranking.md/csv`：候选 checkpoint 排名，包含平均胜率、最差 matchup、死亡高分位、己塔血低分位和超时率。
- `v1.2_candidate_gate.md/csv`：按 v1.2 验收标准逐项判定候选 checkpoint，额外检查 `death_tail_risk`、`self_tower_tail_risk` 和 `timeout_rate`，避免只看均值。
- `evaluation_matrix.md/csv`：固定矩阵评估清单。
- `evaluation_configs/`：平台兼容 TOML、`usr_conf` JSONL，以及把 TOML 文件绑定回 `eval_id/checkpoint/repeat/usr_conf` 的 `toml_metadata.csv/jsonl`。
- `summoner_skill_grid.md/csv`：召唤师技能网格。
- `summoner_skill_results.md/csv`：若有 episode 账本，按技能选择聚合结果。
- `summoner_skill_recommendations.md/csv`：若有技能网格结果，按 checkpoint 与 matchup 推荐候选技能，并标出是否不同于当前规则表。
- `summoner_skill_policy_gate.md/csv`：检查推荐技能相对当前规则表是否有足够局数、胜率增益、死亡和塔血风险对照；通过后再审阅 patch。
- `summoner_skill_policy_patch.md/py`：把推荐结果导出为可审阅的 matchup-conditioned 规则表草案；确认后再手动并入 `agent_ppo/conf/summoner_skill.py`。
- `checkpoint_matrix.md/csv` 与 `checkpoint_elo.csv`：若有历史模型对抗记录，生成胜率矩阵和 Elo。

`matchup_summary` 中的 `eval_ids` 和 `repeat_indices` 用于回溯固定评估矩阵里的具体单局；`death_p90`、`self_tower_hp_p10`、`timeout_rate` 和 `frame_p90` 用于判断风险尾部是否稳定，避免平均死亡或平均胜率掩盖个别 matchup 崩盘。`avg_push_window_active_frames` 和 `avg_unsafe_dive_active_frames` 可以直接用于验证 v1.2 的科研主线：模型是否在更多安全推塔窗口内造成塔伤，并减少无兵线、低血或被塔锁定时的危险越塔。`avg_unsafe_dive_severity` 记录低血、无兵线、塔锁定和敌方近身叠加后的分层越塔风险。`push_window_tower_damage_share` 表示窗口内塔伤占总敌塔掉血的比例，`unsafe_dive_death_corr` 表示 episode 级危险越塔帧数与死亡数的相关性；样本不足或方差为 0 时会留空。

`checkpoint_ranking` 默认只使用 `is_eval=True` 且 `opponent_agent=common_ai` 的 matchup 行，避免训练局、latest selfplay 或历史模型对抗混入 v1.2-a 的固定评估口径。历史模型池和 selfplay 的对抗结果应通过 `checkpoint_matrix.md/csv` 单独分析。

## 4. 固定矩阵评估

先生成 2 个 v1.1 候选 checkpoint 的固定评估矩阵：

```bash
python3 utils/evaluation_matrix.py \
  --checkpoints 15000,17057 \
  --heroes 112,133,199 \
  --repeats 20 \
  --csv logs/v1.2/evaluation_matrix.csv \
  --md logs/v1.2/evaluation_matrix.md
```

导出平台配置片段：

```bash
python3 utils/evaluation_config_export.py \
  logs/v1.2/evaluation_matrix.csv \
  --output-dir logs/v1.2/evaluation_configs
```

注意：

- TOML 文件只包含平台 `train_env_conf.toml` 已知字段。
- `eval_id/checkpoint_step/matchup/repeat_index` 保存在 `eval_usr_conf.jsonl`、`usr_conf["evaluation"]`、文件名和 `toml_metadata.csv/jsonl` 中；训练账本的 `episode_end` 会把 `usr_conf["evaluation"]` 提升到顶层 `evaluation` 字段。
- 如果只能在平台侧提交 TOML 文件，务必同步保留同目录的 `toml_metadata.csv/jsonl`，用于把单局结果追溯回完整 `usr_conf` 与固定评估矩阵行。
- 技能网格评估时，workflow 会保留 TOML 中已有的 `select_skill`，不会用规则表覆盖。

## 5. v1.2-b 切换

只有当 v1.2-a 的 common_ai 结果稳定后，再切更强对手。

历史模型池课程：

```toml
[episode]
opponent_agent = "curriculum"
eval_opponent_type = "common_ai"
```

并在 `kaiwu.json` 中维护历史模型 ID。课程采样会混合 `common_ai`、历史模型和 `selfplay`。

默认课程比例是 `common_ai:historical:selfplay = 4:4:2`。如果需要复现实验比例，可以用环境变量显式覆盖：

```bash
export HOK_OPPONENT_SCHEDULE=common_ai:4,historical:4,selfplay:2
```

其中 `historical` 会均匀展开为 `kaiwu.json` 里的历史模型 ID；如果历史池为空，会自动退回可用的 `common_ai/selfplay` 候选。训练账本会同时记录 `configured_opponent_agent`、实际 `opponent_agent`、`opponent_source`、`opponent_schedule` 和 `model_pool`，证据包里的 `opponent_curriculum_summary.md/csv` 用于复核真实采样比例。

如果只想做单一历史模型对抗：

```toml
[episode]
opponent_agent = "模型ID"
eval_opponent_type = "common_ai"
```

v1.2-b 训练后重新运行证据包命令，并重点查看：

- `checkpoint_matrix.md`
- `checkpoint_elo.csv`
- `opponent_curriculum_summary.md/csv`
- `matchup_summary.md`
- `summoner_skill_results.md`
- `summoner_skill_recommendations.md`
- `summoner_skill_policy_gate.md/csv`

## 6. 提交候选模型的最低证据

候选 checkpoint 至少需要：

- 9 个 matchup 都有统计。
- 每个 matchup 至少有 20 局重复评估，`v1.2_candidate_gate.md` 中 `matchup_min_episodes` 不应低于 20。
- 最差 matchup 胜率没有明显掉队。
- 平均死亡不高于 v1.1 后段。
- 死亡高分位、己塔血低分位和超时率没有显示高风险长尾。
- 敌塔血或胜局帧数有明确改善。
- `push_window_tower_damage_share` 有记录，`avg_unsafe_dive_severity` 和 `unsafe_dive_death_corr` 没有显示危险越塔风险失控。
- `checkpoint_ranking.md` 能解释为什么选它，而不是默认选最后一步。
- `v1.2_candidate_gate.md` 没有 `FAIL` 或 `MISSING` 项。
