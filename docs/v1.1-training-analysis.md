# v1.1 训练分析：结果复盘与问题定位

保存日期：2026-06-22

本文基于 `logs/v1.1/step-*.md`、`logs/v1.1/training-samples-comparison.png`、`docs/dev-guide/*` 的任务定义，以及 v1.1 代码改动进行分析。结论先行：v1.1 的特征、奖励和召唤师技能改动已经让 PPO 学到了可用的推塔/对线策略，但训练后段出现了稳定性不足，且当前评估证据还不足以判断它能否泛化到 9 个英雄组合。

## 1. 任务约束回顾

1v1 任务的最终目标不是击杀，而是摧毁敌方防御塔。文档中明确：胜利方得一分，失败方不得分；达到 20000 帧超时或异常时双方均不得分。英雄池为鲁班七号 `112`、狄仁杰 `133`、公孙离 `199`，智能体可在 `env.reset()` 前根据双方英雄选择召唤师技能。

因此 v1.2 的优化目标仍应按以下优先级排序：

1. 提高累计胜场数。
2. 降低敌方塔血、提高己方塔血。
3. 缩短有效胜局帧数，减少拖延和超时。
4. 在不牺牲推塔的前提下降低死亡和无效战斗。

## 2. v1.1 已完成的关键改动

v1.1 的核心提交为 `31069dd Optimize PPO agent training signals`，主要覆盖以下方向：

- 召唤师技能从随机选择改为规则表：鲁班七号默认闪现，狄仁杰默认狂暴，公孙离默认狂暴，公孙离对狄仁杰时覆盖为净化。
- 特征维度从默认极小特征扩展到 `Config.FEATURE_DIM = 63`。
- 英雄特征加入己方/敌方英雄 ID、存活、血量、能量、等级、经验、经济、攻防、移速、攻速、位置、K/D。
- 技能特征加入前 4 个技能槽的可用性和冷却比例。
- 防御塔特征从只看敌塔扩展为己方塔和敌方塔。
- 新增小兵线汇总特征，包括双方小兵数量、血量和推进位置。
- 奖励权重扩展为推塔、摧毁塔、血量、经济、经验、击杀、死亡、前进。
- 训练 workflow 使用 `[112, 133, 199]` 轮训阵容，并上报 reward detail。

这些改动与 `docs/optimization-pipeline.md` 中第一轮、第二轮建议高度一致：先补信息，再补 dense reward，然后用监控闭环判断。

## 3. v1.1 训练曲线摘要

训练任务配置显示：

- `opponent_agent: selfplay`
- `eval_opponent_type: common_ai`
- `eval_interval: 10`
- 日志采样范围：`step-0` 到 `step-17057`

关键节点如下：

| step | common_ai 胜率 | common_ai 敌塔血 | common_ai 己塔血 | common_ai K/D | selfplay 胜率 | reward |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| 0 | 0.00 | 12000 | 4 | 0.00 / 0.00 | 0.35 | -19.85 |
| 4000 | 0.20 | 9600 | 1726 | 0.80 / 0.00 | 0.50 | -8.70 |
| 7000 | 0.06 | 10674 | 490 | 0.33 / 2.00 | 0.38 | -28.78 |
| 10000 | 0.56 | 4171 | 5431 | 2.66 / 2.59 | 0.56 | -3.65 |
| 12000 | 0.69 | 2906 | 6390 | 3.00 / 2.66 | 0.53 | 1.35 |
| 15000 | 0.84 | 1401 | 7745 | 3.75 / 2.72 | 0.44 | 9.23 |
| 17057 | 0.75 | 1252 | 6690 | 3.41 / 3.09 | 0.53 | 3.15 |

### 主要现象

1. v1.1 不是无效训练。`common_ai` 胜率从 0 提升到峰值 0.84，敌塔剩余血量从 12000 降到最低约 1252，说明模型确实学会了推进、发育和攻击塔。
2. 学习存在明显启动期。0 到 7000 步基本不稳定，7000 步附近出现死亡升高、reward 最低；8000 步后胜率、经济和伤害同步上升。
3. 峰值不在最终 checkpoint。`step-15000` 的 common_ai 胜率最高，为 0.84；最终 `step-17057` 回落到 0.75。虽然最终敌塔血更低，但己塔血也下降，死亡升到 3.09。
4. selfplay 指标解释力有限。早期 `step-2000` selfplay 胜率达到 0.69，但 common_ai 胜率仍为 0，说明 selfplay 胜率可能受双方同弱、换边、采样波动影响，不能单独作为选模依据。
5. 当前日志不足以验证 9 组合泛化。代码已轮训 3 个英雄两两组合，但日志中的 User Config 持续显示 `199 vs 133`，且没有按英雄 matchup 拆分的胜率、塔血、死亡指标。

## 4. 问题定位

### 4.1 训练后段稳定性不足

`step-15000` 到 `step-17057`：

- common_ai 胜率从 0.84 降到 0.75。
- 己方塔血从 7745 降到 6690。
- 死亡从 2.72 升到 3.09。
- reward 从 9.23 降到 3.15。

这更像策略在继续探索或自博弈适配中发生漂移，而不是持续单调改进。v1.2 不宜只“继续多训”，应先加入 checkpoint 选择和训练稳定性控制。

### 4.2 reward 没有直接表达终局胜负和超时

当前 reward 由状态差分构成，包含 `tower_destroy`，但没有显式使用 `observation["win"]` 或终局结果做最终奖励，也没有对超时按塔血差处理。对于“累计胜场数”任务，缺少终局信号会让模型在中后期过度依赖局部指标，例如击杀、前进或经济。

### 4.3 死亡惩罚仍不足以约束高风险打法

common_ai 变强的同时死亡也明显升高。`step-15000` 胜率最高但死亡已达 2.72，`step-17057` 死亡升至 3.09。说明模型可能学到了以高频战斗换取推进的打法。它能打 common_ai，但遇到更强模型时可能被惩罚。

### 4.4 兵线特征过粗，缺少“能否安全点塔”的判断

v1.1 已加入小兵数量、血量、均值推进位置，但缺少：

- 敌塔附近是否有己方兵线。
- 己方英雄是否处于敌塔攻击范围。
- 敌塔当前目标是否为英雄还是小兵。
- 小兵与塔、英雄与塔之间的相对关系。

这些信息直接决定射手 1v1 的推塔窗口。

### 4.5 评估粒度不够

当前监控能看到整体 `win_rate`、塔血、K/D、经济、伤害，但不能回答：

- 哪个英雄组合最弱。
- 哪个召唤师技能选择有效。
- 是蓝方还是红方更强。
- 高胜率 checkpoint 是否在其他 matchup 崩盘。

v1.2 必须先把实验账本补齐，否则容易围绕偶然峰值调参。

## 5. v1.2 总体判断

v1.2 的主线应是“稳定选模 + 终局对齐 + 推塔窗口建模”，而不是大改算法。PPO 本身已经能从 v1.1 信号中学到策略；下一步收益更可能来自奖励对齐、特征补缺、评估拆分和训练课程。

建议把 `step-15000` 作为 v1.1 候选最佳 checkpoint，把 `step-17057` 作为最终 checkpoint 备选；v1.2 开始前先对这两个 checkpoint 做固定矩阵评估。如果只能选一个继续训练，优先从 `step-15000` 开始，因为它的 common_ai 胜率、己塔血、经济和 reward 更一致。
