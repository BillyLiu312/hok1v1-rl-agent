# v1.2 实施计划：从 v1.1 到更稳定的胜率提升

保存日期：2026-06-22

本文承接 `docs/v1.1-training-analysis.md`。v1.2 的目标不是一次性重写 PPO，而是在 v1.1 已经有效的基础上补齐终局信号、推塔窗口特征和评估闭环，使模型从“能打 common_ai”推进到“能稳定赢、能跨英雄组合泛化”。

## 0. v1.2 验收标准

v1.2 的候选模型至少满足：

- common_ai 总胜率高于 v1.1 最佳候选 `step-15000` 的 0.84，或在胜率接近时显著降低死亡。
- 9 个英雄组合都有单独统计，最差组合不低于整体胜率太多。
- 平均敌塔血低于 v1.1 `step-15000` 的 1401，或胜局平均帧数更短。
- 平均死亡低于 v1.1 后段的 3.09，避免学成高风险互换。
- 能明确说明选择哪个 checkpoint，而不是默认采用最终 step。

## 1. P0：先补评估和实验账本

### 1.1 生成完整训练摘要

新增或扩展一个本地日志分析脚本，将 `logs/v*/step-*.md` 汇总成 CSV/Markdown：

- step、actual_train_global_step、episode_cnt、sample_receive_cnt。
- common_ai / selfplay 的 win_rate、self_tower_hp、enemy_tower_hp、frame、money_per_frame、kill、death、hurt_to_hero、hurt_by_hero。
- reward、total_loss、policy_loss、value_loss、entropy_loss。
- reward detail 各项。

当前 `training-samples-comparison.png` 只覆盖到 `step-6196`，不足以反映 v1.1 后半段峰值与回落。v1.2 开始后每次训练都应自动生成完整摘要。

建议文件：

- `utils/analyze_training_logs.py`
- 输出到 `logs/v1.2/summary.md` 和 `logs/v1.2/summary.csv`

### 1.2 按英雄 matchup 拆分监控

在 workflow 上报中加入当前蓝红英雄 ID、monitor_side 英雄 ID、opponent 英雄 ID。若平台指标 label 不便拆分，至少在本地日志中打印结构化行，方便离线解析。

需要回答的问题：

- `112 vs 112`、`112 vs 133`、...、`199 vs 199` 各自胜率是多少。
- 每个 matchup 的死亡、敌塔血、帧数是否异常。
- 召唤师技能覆盖规则是否真的提升了对应 matchup。

建议文件：

- `agent_ppo/workflow/train_workflow.py`
- `agent_ppo/conf/monitor_builder.py`

## 2. P1：奖励对齐终局胜负

### 2.1 增加终局奖励

当前 reward 主要是逐帧差分，缺少直接胜负信号。建议在 episode 结束时根据 `observation[str(i)]["win"]` 或终局塔状态给最后一帧追加奖励：

```text
win_result:
  win      +20.0
  lose     -20.0
  timeout  按塔血差给 [-8.0, 8.0]
```

注意：终局奖励只在最后一帧写入 `frame_collector.save_last_frame()` 或等价路径，避免每帧重复注入。

建议文件：

- `agent_ppo/workflow/train_workflow.py`
- `agent_ppo/feature/reward_process.py`
- `agent_ppo/conf/conf.py`

### 2.2 拆分保塔奖励

当前 `tower_hp_point` 使用双方差分，可以表达推塔和保塔，但调参不够直观。建议拆出：

- `enemy_tower_hp_down`：敌塔掉血正奖励。
- `self_tower_hp_down`：己塔掉血负奖励。
- `tower_destroy`：摧毁敌塔终局大正，己塔被摧毁大负。

这样可以单独观察“高胜率是否靠偷塔牺牲保塔”。

### 2.3 调整死亡风险

v1.1 后段死亡偏高。建议 v1.2 第一轮把死亡权重从 `3.0` 提高到 `4.0`，并加一个低血危险区惩罚：

- 己方血量低于 35% 且敌方英雄存活且距离较近：负奖励。
- 己方低血且正在接近敌塔、且无己方兵线在敌塔附近：更强负奖励。

这不是鼓励怂，而是避免“无兵线越塔送命”。

## 3. P1：补推塔窗口特征

v1.1 的 63 维特征已经覆盖基本信息，但对推塔窗口判断仍粗。建议 v1.2 增加少量高价值特征，控制在 10 到 20 维内。

### 3.1 塔攻击范围与塔目标

新增：

- 己方英雄到敌塔距离。
- 己方英雄是否接近敌塔危险范围。
- 敌塔 `attack_target` 是否为己方英雄。
- 敌塔 `attack_target` 是否为己方小兵。
- 己方塔 `attack_target` 是否为敌方英雄。

建议文件：

- `agent_ppo/feature/feature_process/organ_process.py`
- `agent_ppo/conf/conf.py`
- `tests/test_ppo_optimization.py`

### 3.2 敌塔附近兵线

新增：

- 敌塔附近己方小兵数量。
- 敌塔附近己方小兵血量总和。
- 敌塔附近敌方小兵数量。
- 己方小兵最近敌塔距离。

建议文件：

- `agent_ppo/feature/feature_process/soldier_process.py`

### 3.3 英雄伤害差与经济差

新增：

- 己方对英雄总伤害与敌方对英雄总伤害的差值。
- 己方承伤与敌方承伤差值。
- 当前经济差、等级差。
- 己方复活时间、敌方复活时间。

这些字段在 `docs/dev-guide/protocol.md` 已列出，属于可观测数据。

## 4. P2：训练课程与 checkpoint 选择

### 4.1 v1.1 checkpoint 复评

先固定评估 `step-15000` 和 `step-17057`：

- 每个英雄组合至少 20 局。
- 开启蓝红换边。
- 记录总胜率、最差 matchup、死亡、敌塔血、己塔血、帧数。

若 `step-15000` 的矩阵评估仍领先，应作为 v1.2 起点；若 `step-17057` 在更多组合上更稳，再选择最终 checkpoint。

### 4.2 课程顺序

建议 v1.2 分两段跑：

1. common_ai 对抗短训：验证终局奖励和推塔窗口特征是否能稳定提升，不先引入复杂 selfplay 漂移。
2. selfplay 或历史模型对抗：在 common_ai 稳定后再提升对抗强度。

如果继续 `opponent_agent = "selfplay"`，建议引入历史模型池，而不是永远只打 latest。latest-vs-latest 容易让策略一起漂移，selfplay 胜率也不容易解释。

## 5. P2：PPO 稳定性调参

只有在 P0/P1 完成后再动算法参数。建议先做保守改动：

- 学习率从 `1e-3` 降到 `5e-4` 或 `3e-4`，观察 12k 到 17k 是否仍回落。
- 保持 `GAMMA = 0.995`，因为推塔是长回合目标。
- 熵系数 `BETA_START = 0.025` 暂不增大；若策略后段仍发散，可小幅降低。
- 保持梯度裁剪 `0.5`。

不要在同一轮同时大改 reward、特征和 PPO 参数；否则无法判断收益来源。

## 6. 推荐实施顺序

1. 新增训练日志汇总脚本，复盘 v1.1 的 `step-15000` 与 `step-17057`。
2. workflow 上报 matchup 信息，建立 9 组合评估表。
3. 加终局胜负/超时奖励，并拆分推塔与保塔奖励。
4. 增加塔目标、塔危险区、敌塔附近兵线特征。
5. 跑 v1.2-a：common_ai 短训，确认 reward 曲线和胜率是否稳定超过 v1.1。
6. 跑 v1.2-b：selfplay 或历史模型对抗，观察泛化和死亡是否恶化。
7. 用矩阵评估选 checkpoint，不默认选最后一个模型。

## 7. 第一轮代码改动清单

最小可执行 v1.2-a 建议只改这些：

- `utils/analyze_training_logs.py`：解析 `logs/v*/step-*.md`。
- `agent_ppo/workflow/train_workflow.py`：终局奖励注入、matchup 日志。
- `agent_ppo/feature/reward_process.py`：新增终局/保塔/危险区 reward 子项。
- `agent_ppo/conf/conf.py`：新增 reward 权重，更新 feature dim。
- `agent_ppo/feature/feature_process/organ_process.py`：塔目标和危险区特征。
- `agent_ppo/feature/feature_process/soldier_process.py`：敌塔附近兵线特征。
- `tests/test_ppo_optimization.py`：覆盖新增 reward key 和 feature length。

第一轮先不要改模型结构。v1.1 已证明现有 PPO 网络容量足够学到 0.84 common_ai 峰值，v1.2 的首要问题是让这个峰值更稳、更可解释。
