# v1.2 实施计划：从 v1.1 到更稳定的胜率提升

保存日期：2026-06-22

本文承接 `docs/v1.1-training-analysis.md`。v1.2 的目标不是一次性重写 PPO，而是在 v1.1 已经有效的基础上补齐终局信号、推塔窗口特征和评估闭环，使模型从“能打 common_ai”推进到“能稳定赢、能跨英雄组合泛化”。

## 0. v1.2 验收标准

v1.2 的候选模型至少满足：

- common_ai 总胜率高于 v1.1 最佳候选 `step-15000` 的 0.84，或在胜率接近时显著降低死亡。
- 9 个英雄组合都有单独统计，最差组合不低于整体胜率太多。
- 平均敌塔血低于 v1.1 `step-15000` 的 1401，或胜局平均帧数更短。
- 平均死亡低于 v1.1 后段的 3.09，避免学成高风险互换。
- 能明确说明选择哪个 checkpoint，而不是默认采用最终 step。

## 1. P0：先补评估和实验账本

### 1.1 生成完整训练摘要

新增或扩展一个本地日志分析脚本，将 `logs/v*/step-*.md` 汇总成 CSV/Markdown：

- step、actual_train_global_step、episode_cnt、sample_receive_cnt。
- common_ai / selfplay 的 win_rate、self_tower_hp、enemy_tower_hp、frame、money_per_frame、kill、death、hurt_to_hero、hurt_by_hero。
- reward、total_loss、policy_loss、value_loss、entropy_loss。
- reward detail 各项。

当前 `training-samples-comparison.png` 只覆盖到 `step-6196`，不足以反映 v1.1 后半段峰值与回落。v1.2 开始后每次训练都应自动生成完整摘要。

建议文件：

- `utils/analyze_training_logs.py`
- 输出到 `logs/v1.2/summary.md` 和 `logs/v1.2/summary.csv`

### 1.2 按英雄 matchup 拆分监控

在 workflow 上报中加入当前蓝红英雄 ID、monitor_side 英雄 ID、opponent 英雄 ID。若平台指标 label 不便拆分，至少在本地日志中打印结构化行，方便离线解析。

需要回答的问题：

- `112 vs 112`、`112 vs 133`、...、`199 vs 199` 各自胜率是多少。
- 每个 matchup 的死亡、敌塔血、帧数是否异常。
- 召唤师技能覆盖规则是否真的提升了对应 matchup。

建议文件：

- `agent_ppo/conf/monitor_builder.py`

## 2. P1：奖励对齐终局胜负

### 2.1 增加终局奖励

当前 reward 主要是逐帧差分，缺少直接胜负信号。建议在 episode 结束时根据 `observation[str(i)]["win"]` 或终局塔状态给最后一帧追加奖励：

```text
win_result:
  win      +20.0
  lose     -20.0
  timeout  按塔血差给 [-8.0, 8.0]
```

注意：终局奖励只在最后一帧写入 `frame_collector.save_last_frame()` 或等价路径，避免每帧重复注入。

建议文件：

- `agent_ppo/workflow/train_workflow.py`
- `agent_ppo/feature/reward_process.py`
- `agent_ppo/conf/conf.py`

### 2.2 拆分保塔奖励

当前 `tower_hp_point` 使用双方差分，可以表达推塔和保塔，但调参不够直观。建议拆出：

- `enemy_tower_hp_down`：敌塔掉血正奖励。
- `self_tower_hp_down`：己塔掉血负奖励。
- `tower_destroy`：摧毁敌塔终局大正，己塔被摧毁大负。

这样可以单独观察“高胜率是否靠偷塔牺牲保塔”。

### 2.3 调整死亡风险

v1.1 后段死亡偏高。建议 v1.2 第一轮把死亡权重从 `3.0` 提高到 `4.0`，并加一个低血危险区惩罚：

- 己方血量低于 35% 且敌方英雄存活且距离较近：负奖励。
- 己方低血且正在接近敌塔、且无己方兵线在敌塔附近：更强负奖励。

这不是鼓励怂，而是避免“无兵线越塔送命”。

- 敌塔 `attack_target` 是否为己方英雄。
- 敌塔 `attack_target` 是否为己方小兵。
- 己方塔 `attack_target` 是否为敌方英雄。

建议文件：

- `agent_ppo/feature/feature_process/organ_process.py`
- `agent_ppo/conf/conf.py`
- `tests/test_ppo_optimization.py`

### 3.2 敌塔附近兵线

新增：

- 敌塔附近己方小兵数量。
- 敌塔附近己方小兵血量总和。
- 敌塔附近敌方小兵数量。
- 己方小兵最近敌塔距离。

建议文件：

- `agent_ppo/feature/feature_process/soldier_process.py`

### 3.3 英雄伤害差与经济差

新增：

这些字段在 `docs/dev-guide/protocol.md` 已列出，属于可观测数据。

## 4. P2：训练课程与 checkpoint 选择

### 4.1 v1.1 checkpoint 复评

先固定评估 `step-15000` 和 `step-17057`：

- 每个英雄组合至少 20 局。
- 开启蓝红换边。
- 记录总胜率、最差 matchup、死亡、敌塔血、己塔血、帧数。

若 `step-15000` 的矩阵评估仍领先，应作为 v1.2 起点；若 `step-17057` 在更多组合上更稳，再选择最终 checkpoint。

### 4.2 课程顺序

建议 v1.2 分两段跑：

1. common_ai 对抗短训：验证终局奖励和推塔窗口特征是否能稳定提升，不先引入复杂 selfplay 漂移。
2. selfplay 或历史模型对抗：在 common_ai 稳定后再提升对抗强度。

如果继续 `opponent_agent = "selfplay"`，建议引入历史模型池，而不是永远只打 latest。latest-vs-latest 容易让策略一起漂移，selfplay 胜率也不容易解释。

当前代码包将 `agent_ppo/conf/train_env_conf.toml` 的默认 `opponent_agent` 设为 `common_ai`，对应 v1.2-a；v1.2-b 再显式切到 `curriculum`、`selfplay` 或历史模型 ID。

## 5. P2：PPO 稳定性调参

只有在 P0/P1 完成后再动算法参数。建议先做保守改动：

- 学习率从 `1e-3` 降到 `5e-4`，作为首轮保守稳定性改动；若仍回落，再评估 `3e-4`。
- 保持 `GAMMA = 0.995`，因为推塔是长回合目标。
- 熵系数 `BETA_START = 0.025` 暂不增大；若策略后段仍发散，可小幅降低。
- 保持梯度裁剪 `0.5`。

不要在同一轮同时大改 reward、特征和 PPO 参数；否则无法判断收益来源。

## 6. P2：能讲科研故事的升级点

v1.2 除了工程涨分，还应保留一条能汇报、能写实验对比的研究主线。建议把故事组织为：

> 在长回合、部分可观测、目标稀疏的 MOBA 1v1 任务中，智能体如何从局部对线行为学到稳定推塔胜利策略。

下面这些升级点都能落到现有 PPO 框架中，不需要重写系统，但每个都能提出清晰假设并做消融。

### 6.1 推塔窗口识别：从状态特征到战术机会建模

问题：v1.1 已能推塔，但死亡偏高，说明模型没有稳定区分“可安全点塔”和“无兵线越塔送命”。

方法：

- 构造 `push_window` 特征：敌塔附近己方小兵数量、敌塔目标是否为小兵、己方到敌塔距离、敌方英雄是否存活且接近。
- 构造 `unsafe_dive` 特征：低血量、无己方兵线、敌塔攻击目标为己方英雄、敌方英雄可威胁。
- reward 中加入“推塔窗口内攻击塔”正奖励和“非窗口越塔承伤”负奖励。
- 额外记录 0 权重诊断量 `push_window_active` / `unsafe_dive_active`，用于统计安全推塔窗口和危险越塔出现频次，不直接改变优化目标。

科研假设：

- 显式推塔窗口特征能提升胜率，同时降低死亡率。
- 相比单纯提高死亡惩罚，窗口建模能保留进攻性，不会把策略推向保守拖延。

验证指标：

- 胜率、死亡、敌塔血、胜局平均帧数。
- `push_window` 内敌塔掉血占比。
- `unsafe_dive` 出现次数与死亡的相关性。
- `avg_push_window_active_frames`、`avg_unsafe_dive_active_frames` 在 9 组合上的分布。

消融实验：

- baseline v1.1。
- 只加死亡惩罚。
- 加推塔窗口特征。
- 加推塔窗口特征 + 对应 reward。
- 使用 `HOK_REWARD_PROFILE=no_window_reward` 和 `HOK_REWARD_PROFILE=no_terminal_reward` 做短训消融，避免手改代码导致实验不可复现。

### 6.2 风险敏感奖励：从平均胜率到稳定胜率

问题：`step-15000` common_ai 胜率最高，但后段回落且死亡升高。只优化平均 reward 可能偏向高方差打法。

方法：

- 把 reward 分成收益项和风险项：推塔收益、经济收益、死亡风险、己塔损失、危险区承伤。
- 对低血量、己塔低血、敌方英雄近身等状态使用更强风险惩罚。
- 在选模时加入“最差 matchup 胜率”和“死亡分位数”，而不只看平均胜率。

科研假设：

- 风险敏感 reward 会牺牲少量极端进攻收益，但提高跨 matchup 的稳定性。
- 最差 matchup 胜率比整体胜率更适合作为多英雄 1v1 的泛化指标。

验证指标：

- 9 组合平均胜率与最低胜率。
- 死亡均值和高分位死亡。
- 己塔血均值和低分位己塔血。
- 超时率。

消融实验：

- 普通 reward。
- 普通 reward + 更高 death 权重。
- 风险分层 reward。

### 6.3 对手课程学习：从 common_ai 拟合到策略泛化

问题：selfplay 胜率早期和 common_ai 胜率不一致，说明单一 selfplay 指标解释力弱；latest-vs-latest 还可能导致策略共同漂移。

方法：

- 课程分三层：common_ai 预热、v1.1/v1.2 历史 checkpoint 对抗、latest selfplay 微调。
- 历史对手池按固定比例采样，例如 `common_ai:historical:selfplay = 4:4:2`，并通过 `HOK_OPPONENT_SCHEDULE` 固定实验比例。
- 每轮训练保留固定评估矩阵，不让训练对手和评估对手混在一起。

科研假设：

- 历史对手池能降低 selfplay 漂移，提高对不同策略的鲁棒性。
- 课程对手比单一 common_ai 或单一 latest selfplay 更能提升最差 matchup 胜率。

验证指标：

- common_ai、历史 checkpoint、latest selfplay 三类评估胜率。
- checkpoint 间 Elo/胜率矩阵。
- 训练后段胜率回落幅度。

消融实验：

- common_ai only。
- latest selfplay only。
- common_ai -> latest selfplay。
- common_ai -> historical pool -> latest selfplay。

### 6.4 条件化召唤师技能策略：从规则表到 matchup-conditioned policy

问题：v1.1 已把召唤师技能从随机改为规则表，但规则表本身还没有实验证明，也无法说明不同 matchup 下为什么有效。

方法：

- 保留当前规则表作为 baseline。
- 对 9 个英雄组合做召唤师技能网格评估，形成 matchup-skill 胜率矩阵。
- 训练后输出 `summoner_skill_recommendations.md/csv`，把网格结果转成 matchup-conditioned 规则表候选。
- 若平台允许，在 `init_config()` 中按 matchup 使用最佳技能；若要更有研究味，可以把“技能选择”视作 episode-level 条件策略。

科研假设：

- 召唤师技能选择是 1v1 策略分布的重要条件变量。
- matchup-conditioned 技能选择能减少训练噪声，并提升最差 matchup。

验证指标：

- 每个 matchup 下不同技能的胜率、死亡、平均帧数。
- 固定技能 vs 规则表 vs matchup-conditioned 表。

实现优先级：

- v1.2 先做网格评估和规则表修正，不急着把技能选择并入神经网络。

### 6.5 可解释训练账本：从平台曲线到可复现实验

问题：v1.1 复盘依赖平台采样 markdown，缺少按 episode 的原始实验账本，导致很难解释峰值、回落和 matchup 差异。

方法：

- 使用 `utils/training_recorder.py` 记录 episode、learner、config、sample、checkpoint 事件。
- 训练后生成 matchup 表、checkpoint 表、reward 分解表。
- 在每次 v1.2 实验文档中固定记录假设、改动、对照组、指标和结论，并记录 `HOK_REWARD_PROFILE` / `HOK_REWARD_WEIGHT_OVERRIDES`。

科研假设：

- 更细粒度的训练账本能把“调参”转化为可复现实验，使每次改动都能被消融验证。

验证指标：

- 每个结论是否能追溯到 JSONL 记录。
- 每次实验是否有明确对照 checkpoint。
- 是否能复现 `step-15000` 优于最终 step 的选模判断。

## 7. 推荐实施顺序

1. 新增训练日志汇总脚本，复盘 v1.1 的 `step-15000` 与 `step-17057`。
2. workflow 上报 matchup 信息，建立 9 组合评估表。
3. 加终局胜负/超时奖励，并拆分推塔与保塔奖励。
4. 增加塔目标、塔危险区、敌塔附近兵线特征。
5. 增加 `push_window` / `unsafe_dive` 统计，验证推塔窗口假设。
6. 跑 v1.2-a：common_ai 短训，确认 reward 曲线和胜率是否稳定超过 v1.1。
7. 跑 v1.2-b：历史模型池或 selfplay 对抗，观察泛化和死亡是否恶化。
8. 做召唤师技能 matchup 网格评估，修正规则表。
9. 用矩阵评估选 checkpoint，不默认选最后一个模型。

## 8. 第一轮代码改动清单

最小可执行 v1.2-a 建议只改这些：

- `agent_ppo/workflow/train_workflow.py`：终局奖励注入、matchup 日志。
- `agent_ppo/feature/reward_process.py`：新增终局/保塔/危险区 reward 子项。
- `agent_ppo/conf/conf.py`：新增 reward 权重，更新 feature dim。
- `agent_ppo/conf/conf.py`：首轮学习率降到 `5e-4`，保持 `GAMMA/BETA/clip` 不变。
- `agent_ppo/conf/train_env_conf.toml`：v1.2-a 默认 `opponent_agent = "common_ai"`，先隔离验证 reward/feature 改动。
- `agent_ppo/feature/feature_process/organ_process.py`：塔目标和危险区特征。
- `agent_ppo/feature/feature_process/soldier_process.py`：敌塔附近兵线特征。
- `tests/test_ppo_optimization.py`：覆盖新增 reward key 和 feature length。

第一轮先不要改模型结构。v1.1 已证明现有 PPO 网络容量足够学到 0.84 common_ai 峰值，v1.2 的首要问题是让这个峰值更稳、更可解释。
